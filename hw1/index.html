git config --global user.name - Це встановить наше ім'я користувача Sarah Smith. Усі наші майбутні коміти посилатимуться на цю інформацію. Ми використовували параметр –global, щоб застосувати цю конфігурацію git за умовчанням до всіх репозиторій, що належать нашому користувачу

git config --global user.email - Щоб настроїти адресу електронної пошти Git, виконайте команду git config –global user.email. Ця команда електронної пошти git config приймає один аргумент: ваша електронна адреса.

git init - Команда git init створює новий репозиторій Git. З її допомогою можна перетворити існуючий проект без керування версіями на репозиторій Git або ініціалізувати новий порожній репозиторій.

git status -Команда git status відображає стан робочого каталогу та розділу проіндексованих файлів. З її допомогою можна перевірити індексацію змін та побачити файли, які не відстежуються Git. Інформація про історію комітів проекту не відображається під час виведення даних про стан. Для цього використається команда git log.

git add-Команда git add додає зміну з робочого каталогу розділ проіндексованих файлів. Вона повідомляє Git, що ви хочете включити зміни до конкретного файлу в наступний коміт. Однак насправді команда git add не істотно впливає на репозиторій: зміни реєструються в ньому тільки після виконання команди git commit

git commit - Коміт проіндексованого стану коду. Ця команда відкриє текстовий редактор із пропозицією ввести коментар до комміту. Після введення коментаря збережіть файл і закрийте текстовий редактор, щоб виконати коміт.

git log - Показує журнали комітів.Перелічіть коміти, які доступні, перейшовши за батьківськими посиланнями з даного(их) коміту(ів), але виключіть коміти, які доступні з того(их), наданого(их) із ^ перед ними. Вихідні дані за замовчуванням надаються у зворотному хронологічному порядку.

git checkout - На цій сторінці розглядається команда git checkout, включаючи приклади використання та прикордонні випадки. У Git під терміном checkout мають на увазі перемикання між різними версіями цільового об'єкта. Команда git checkout працює з трьома різними об'єктами: файлами, коммітами та гілками. Під перемиканням також зазвичай розуміють дію, пов'язану з виконанням команди git checkout.

git branch - це команда для керування гілками в репозиторії Git. Гілка в Git'і — це просто «ковзкий» покажчик на один із коммітів. Коли ви створюєте нові комміти, покажчик гілки автоматично зсувається вперед до новоствореного комміту. Гілка, що створюється першою в новому репозиторії, за замовчуванням називається master

git push - Команда git push використовується для вивантаження вмісту локального репозиторію у віддалений репозиторій. Вона дозволяє передати коміти з локального репозиторію у віддалений. Ця команда симетрична команді git fetch: при вилученні за допомогою fetch коміти імпортуються в локальні гілки, а при публікації за допомогою push коміти експортуються у віддалені гілки. Налаштувати віддалені гілки можна за допомогою команди git remote. Команда push може перезаписати зміни, тому при її використанні слід бути обережними.

git pull - використовується для вилучення та завантаження вмісту з віддаленого репозиторію та негайного оновлення локального репозиторію цим вмістом. Злиття віддалених вищих змін у локальний репозиторій - це звичайне завдання робочого процесу, що виникає при спільній роботі на основі системи Git. Команда git pull насправді є комбінацією двох інших команд: git fetch і git merge. На першому етапі git pull виконується команда git fetch, обмежена локальною гілкою, яку вказує HEAD. Відразу після завантаження вмісту команда git pull виконує злиття. Для злитого вмісту створюється новий коміт, а покажчик HEAD оновлюється і починає вказувати цей новий коміт

git merge - Команда git merge поєднує кілька послідовностей коммітів у загальну історію. Найчастіше команду git merge використовують для об'єднання двох гілок

git clone - клонувати сховище, але без можливості редагувати файли. Це включає посилання, або гілки. Ви можете використовувати це, якщо ви намагаєтеся створити вторинну копію репозиторію на окремому віддаленому пристрої та хочете зіставити всі гілки. Це може статися під час налаштування за допомогою нового пульта дистанційного керування для вашого хостингу Git або під час використання Git під час автоматичного тестування

git revert - Можна сказати, що команда git revert є типовою командою відміни. Однак принцип її дії відрізняється від первинної зміни змін. Замість видалення коміту з історії проекту ця команда змінює внесені в ньому зміни та додає новий комміт з отриманим вмістом. У результаті історія в Git не теряється, що важливо для забезпечення цілої версії історії та надійної спільної роботи.

git reset - це складний універсальний інструмент для зміни змін. Вона має три основні форми виклику, відповідні аргументи командної строки --soft, --mixed, --hard. Кожен із цих трьох аргументів відповідає трем внутрішнім механізмам керування станом Git: дереву комітів (HEAD), розділу проіндексованих файлів і робочому каталогу

git rebase - Перебазування — це процес переміщення послідовності комітів до нового базового коміту або їх об'єднання. Операцію перебазування зручно всього застосувати та відобразити в контексті створення функціональних веток

git cherry-pick - це потужна команда, яка дозволяє вибирати довільні комміти Git за посиланням і додавати їх до поточного робочого HEAD. Збір вишень – це процес вибору комміту з гілки та застосування його до іншої. git cherry-pickможе бути корисним для скасування змін. Наприклад, припустимо, що коміт випадково зроблено не в ту гілку. Ви можете перейти на правильну гілку і вибрати коміт туди, де він має бути.

git stash - Команда git stash дозволяє на якийсь час «здати в архів» (або відкласти) зміни, зроблені в робочій копії, щоб ви могли застосувати їх пізніше. Відкладення змін корисне, якщо вам необхідно переключити контекст і ви поки що не готові до створення комміту
